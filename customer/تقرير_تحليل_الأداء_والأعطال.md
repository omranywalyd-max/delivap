## 1. ملخص تنفيذي

هذا التقرير يغطي **36 مشكلة** في تطبيق الزبون، مصنفة كالتالي:

| التصنيف | 🔴 كراش | 🟡 تسريب ذاكرة | 🟢 أداء/صورة | 🔵 شبكة |
|---------|---------|----------------|-------------|---------|
| العدد | 12 | 9 | 10 | 5 |

**أخطر 5 مشاكل يجب معالجتها فوراً:**

1. **🔴 `stores_view.dart:590`** — `setState()` بعد `await _uploadImage()` بدون `mounted` — **كراش مضمون** لو المستخدم طلع من الشيت قبل اكتمال الرفع
2. **🔴 `profile_screen.dart:1381`** — `_snack()` تستعمل `ScaffoldMessenger.of(context)` بدون `mounted` بعد async — **كراش** لو المستخدم ضغط Back أثناء حفظ التعديلات
3. **🔴 `stores_view.dart:617`** — `Navigator.pop(context)` بعد async — `mounted` ما تحميش من هاد الحالة لأن الـ route ممكن تكون راحت
4. **🟡 5 static Maps unbounded** — `StoreColorCache`, `_storeDataCache`, `_ItemColorCache`, `_ProdCache`, `_DrinksCache` — تنمو بلا حدود وتستهلك RAM طول عمر التطبيق
5. **🟢 17+ `CachedNetworkImage` بدون `memCacheWidth`** — الصور تُفك بكامل دقتها (4000×3000+) في الذاكرة — خطر OOM على الأجهزة الضعيفة

#### 3.1 `stores_view.dart:590` — كراش بعد رفع الصورة

```dart
590:         setState(() => _isAdding = false);   // ← كراش هنا
591:       }
```

**الكود بعد (الإصلاح)**:
```dart
587:       imageUrl = await _uploadImage(_selectedImage!);
588:       if (!mounted) return;
589:       if (imageUrl.isEmpty) {
590:         _snack('فشل رفع الصورة، تأكد من الاتصال', isError: true);
```

**كيفية الاختبار**:

#### 3.3 `stores_view.dart:617` — `Navigator.pop` بعد async والـ context راح

**السبب الجذري**:  
`mounted` في السطر 615 ما يضمنش أن الـ route لسا في الـ Navigator stack. لو المستخدم طلع من الشيت بطريقة تانية (مثلاً السوائب Down يسبق callback) `Navigator.pop(context)` يرمي `"Looking up a deactivated widget's ancestor is invalid"`.

**الكود قبل**:
```dart
615:     if (mounted) {
616:       Navigator.pop(context);     // ← يمكن يرمي حتى لو mounted true
617:       _snack('✅ تمت الإضافة للسلة');
```

**الكود بعد**:
```dart
615:     if (mounted && context.mounted) {
616:       Navigator.maybePop(context);
617:     }
618:     if (mounted) {
619:       _snack('✅ تمت الإضافة للسلة');
```

**ملاحظة**: `context.mounted` متاح من Flutter 3.7+.

**السبب الجذري**:  
`GridView.builder` يبني `CategoryCardWidget` بدون explicit `Key`. لما تتغير الـ categories (بعد fetch)، Flutter يطابق الأطفال حسب الـ index مش identity، مما يسبب إعادة بناء كل الكروت وفقدان حالة الـ AnimatedWidget داخلها.

**الحل**:
```dart
itemBuilder: (context, i) {
  final cat = _categories[i];
  return CategoryCardWidget(
    key: ValueKey(cat.id),
    ...
  );
}
```

### 🔵 مجموعة 5: الشبكة

#### 3.18 `/api/stores` بدون pagination (`dashboard_screen.dart:165`)

**السبب الجذري**:  
`ApiClient.getList('/api/stores')` يجيب كل المحلات في طلب واحد. لو المحلات وصلت 200-300 محل، هذا يعني JSON كبير (يمكن 1-2MB) يستهلك RAM + bandwidth + CPU للـ parse. يحتاج إضافة `?limit=50` في السيرفر والـ client.

**الحل المرحلي** (Client + Server):
- السيرفر: `router.get('/api/stores', async (req, res) => { const { limit = 50, offset = 0 } = req.query; ... })`
- العميل: ما يزيدش لأننا مررنا `stores` عبر constructor — التحسين يتم في السيرفر

---

#### 3.19 `apiTimeout: 10s` قصير جداً (`env_config.dart:5`)

**السبب الجذري**:  
`static const Duration apiTimeout = Duration(seconds: 10);` — 10 ثواني تكفي للطلبات العادية، لكن للـ upload (صور حتى 1MB بعد compression) قد لا تكفي. الـ `upload()` بالفعل عنده timeout منفصل (60s في السطر 237)، باقي الـ API calls في `api_client.dart` تستعمل الـ 10s العام.

**الحل**:
```dart
static const Duration apiTimeout = Duration(seconds: 30);
```

---

#### 3.20 Socket.io reconnect 5 محاولات فقط (`socket_client.dart:21`)

**السبب الجذري**:  
`'reconnectionAttempts': 5` — بعد 5 محاولات فاشلة (مع تأخير 2s-10s)، السوكيت يتوقف عن محاولة إعادة الاتصال. في تطبيق توصيل، اتصال السوكيت أساسي للـ real-time updates. المحاولات الخاصة (5) تنفذ في ~20s كحد أقصى — لو الشبكة ضعيفة وقطعت 20s+، السوكيت ينقطع للأبد حتى يعيد المستخدم تشغيل التطبيق.

**الحل**:
```dart
'reconnectionAttempts': 20,
'reconnectionDelay': 1000,    // أسرع reconnection
```

## 6. أشياء تحتاج مراقبة إضافية

هذه مشاكل مشكوك فيها لكن تحتاج بيانات من Crashlytics/تحليل قبل تأكيدها:

| المشكلة | الملف | ما نحتاج لتأكيدها |
|---------|-------|------------------|
| **`ImageCache` limits** | Flutter default (1000 images, 50MB) | هل لاحظنا OOM في Crashlytics على أجهزة 2-3GB RAM؟ إذا آه، نحتاج نضبط `ImageCache.maximumSize` و `maximumSizeBytes` |
| **`_loadCategories` بعد dispose** | `stores_view.dart:164` | هل تظهر تقارير عن network calls بعد ما المستخدم طلع من الشاشة؟ |
| **Socket.io reconnect storms** | `socket_client.dart` | هل تظهر logs عن "reconnect_error" متكررة؟ العدد الكبير يمكن يسبب request flood |
| **`AnimationController` في `initState` قبل attach** | `profile_screen.dart:100` | هل تظهر warnings في logs عن `AnimationController.forward()` في `initState`؟ |
| **ProGuard/R8 keep rules** | `proguard-rules.pro:62` | التأكد أن كل الـ model classes اللي تمر عبر JSON `fromJson`/`toJson` محفوظة — إذا استعمل `dart:convert` (مش `json_serializable`)، مشكلة أقل. إذا استعمل `json_serializable` و `@JsonKey`، بعض الـ classes يمكن تنضغط في release build وتسبب كراش |
| **`minSdk: 26`** | `android/app/build.gradle.kts:26` | هذا يعني Android 8.0+ فقط. إذا كان 95%+ من المستخدمين 8.0+، لا مشكلة. إذا في مستخدمين على 7.0- (Android 7 = 4-5% globally)، يحتاج تعديل |

**طلب**:  
إذا عندك تقارير Crashlytics (Firebase Crash Reporting أو Sentry)، ممكن ترسلي أبرز 5-10 كراشات عشان أقدر أطابقها مع الكود ونحدد الأولويات بدقة أكبر.
